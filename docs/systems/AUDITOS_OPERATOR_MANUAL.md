## 1. نظرة عامة على المنتج (Product Overview)

### ما هو AuditOS؟

نظام تدقيق مالي مؤسسي يغطي دورة حياة التدقيق القانوني الكاملة — من استلام ميزان المراجعة حتى اعتماد ونشر القوائم المالية — مع تتبّع كامل، مساعدة ذكاء اصطناعي، ومراجعة واعتماد قائمين على الأدوار.

### لمن صُمم؟

مكاتب التدقيق التي تجري تدقيقات IFRS for SMEs وما يشابهها. صُمم لفرق التدقيق المكونة من مُعدّين، مراجعين، وشركاء يحتاجون إلى تعاون منظم وقابل للتدقيق.

### المشاكل التي يحلها:

- تشتت جداول البيانات وسير العمل عبر البريد الإلكتروني
- غياب التتبّع من البيانات المصدرية إلى القوائم المنشورة
- صياغة يدوية لإيضاحات الإفصاح المعيارية
- عدم وجود ربط منظم بين الأدلة والملاحظات
- اعتماد دون فحوصات جاهزية

### المبدأ الأساسي:

> **الذكاء الاصطناعي يساعد. البشر يقررون. الأدلة تحكم.**
>
> كل مخرجات الذكاء الاصطناعي هي مسوّدة وتحتاج مراجعة بشرية. لا يصبح أي محتوى نهائياً تلقائياً. النظام يفرض هذا الفصل على كل المستويات.

### النطاق الإجمالي:

| المكوّن | الحالة |
|----------|--------|
| مساحة العمل المحكومة | `/audit` |
| العرض التوضيحي الموجّه | `/auditos` |
| ٢١ workflow مدعوم | ✅ |
| ٩ workflows مستبعدة | مؤجلة لما بعد البايلوت |

## 3. القيود المعروفة (Known Limitations)

| القيد | التأثير | التعامل |
|--------|--------|----------|
| **JSON فقط للتصدير** | لا يوجد PDF أو Word حالياً | استخدم JSON مع أدوات تحويل خارجية |
| **لا فحص فيروسات** | الملفات المرفوعة لا تُفحص | استخدم ملفات موثوقة فقط في البايلوت |
| **مصادقة غير مكتملة للإنتاج** | إنشاء المستخدمين يدوي | تجهيز المستخدمين عبر Prisma Studio أو الـ seed |
| **عزل المستأجرين غير مختبَر** | لم يتم التحقق لمنظمتين | بايلوت لمنظمة واحدة فقط |
| **لا تحكم تزامني** | تعديلات متزامنة قد تتعارض | بايلوت لسيناريوهات مستخدم واحد |
| **حدود الملفات الكبيرة** | TB > 50MB أو > 50,000 سطر قد يسبب بطء | استخدم CSV للملفات الكبيرة |
| **pg_dump خارج PATH** | نسخ احتياطي يدوي فقط | استخدم `npm run db:backup` |
| **لا تسجيل ذاتي** | الحسابات تُنشأ يدوياً | استخدم seed أو Prisma Studio |

### ما هو آمن للبايلوت:
- بيانات اصطناعية متحكم بها ✅
- فريق داخلي أو أصحاب مصلحة مدعوّين ✅
- جولة موجهة بسيناريوهات معروفة ✅
- لا ملفات عملاء حقيقية حساسة ✅

## 4. تشغيل البيئة المحلية (How to Start Local Environment)

### المتطلبات الأساسية:
- Node.js (v20+)
- PostgreSQL (v16+) يعمل على `localhost:5432`
- قاعدة بيانات باسم `aqliya`

### خطوات التشغيل:

```bash
# 1. تثبيت الاعتماديات
npm install

# 2. توليد Prisma Client
npx prisma generate

# 3. تطبيق الترحيلات على قاعدة البيانات
npx prisma migrate dev

# 4. تحميل بيانات البايلوت
npm run seed:audit

# 5. تشغيل خادم التطوير
npm run dev
```

يفتح التطبيق على `http://localhost:3001`

### فحص الصحة:
```bash
npm run audit:health    # فحص صحة AuditOS
npm run backup:verify   # فحص سلامة النسخ الاحتياطي
```

### أوامر مفيدة:
```bash
npm run db:studio       # فتح Prisma Studio (واجهة رسومية لقاعدة البيانات)
npm run db:backup       # نسخ احتياطي لقاعدة البيانات
npm run db:restore      # استعادة النسخة الاحتياطية
npm run build           # بناء إنتاجي
npm run lint            # فحص الكود
npm test                # تشغيل الاختبارات
```

## 5. تدفق الدخول (Login Flow)

### بيانات الدخول الافتراضية:

| البريد الإلكتروني | كلمة المرور | الدور | الصلاحية |
|-------------------|-------------|--------|-----------|
| `admin@aqliya.com` | `admin123` | ADMIN | كامل — الإعداد والتهيئة |
| `sara@aqliya.com` | `operator123` | OPERATOR | إعداد الارتباطات — رفع، إنشاء، ربط |
| `mohammad@aqliya.com` | `viewer123` | VIEWER | عرض فقط — لوحة التحكم والتقارير |

> هذه الحسابات من `prisma/seed.ts` (جدول `User` لنظام DecisionOS).  
> حسابات نظام AuditOS من `prisma/seed-audit.ts` (جدول `AuditUser`) لا تحتوي على كلمة مرور — تُستخدم داخل سياق الجلسة بعد الدخول.

### مسار الدخول:

1. افتح `http://localhost:3001/login`
2. أدخل البريد الإلكتروني وكلمة المرور
3. في حال نجاح الدخول، ستُوجّه إلى الصفحة الرئيسية `/`
4. مساحة عمل AuditOS: `/audit`
5. مساحة العرض التوضيحي (بدون تسجيل دخول): `/auditos`

### حالات الخطأ:
- "بريد إلكتروني أو كلمة مرور غير صحيحة" — تحقق من صحة البيانات
- "تم تسجيل الدخول ولكن الجلسة لم تُنشأ" — أعد المحاولة
- إذا رأيت صفحة "Access Denied" — دورك لا يملك الصلاحية الكافية

### عرض توضيحي بدون تسجيل دخول:
المسار `/auditos` مفتوح للعامة ويستخدم بيانات وهمية. مناسب للعروض السريعة دون الحاجة لتسجيل الدخول.

## 7. دليل تشغيل كامل لسير العمل (Full Workflow Operating Guide)

### 7.1 Engagement — إنشاء وإدارة الارتباط

**المسار:** `/audit/engagements/[engagementId]`

1. من لوحة التحكم `/audit`، اختر ارتباطاً موجوداً
2. أو أنشئ ارتباطاً جديداً من خلال واجهة الإدارة
3. كل ارتباط يحتوي على:
   - معلومات العميل
   - الفترة المالية
   - الإطار المحاسبي
   - فريق العمل
   - شريط تقدّم workflow

**حالات الارتباط:** setup → in_progress → under_review → ready_for_approval → approved → published

---

### 7.2 Trial Balance — استيراد ميزان المراجعة

**المسار:** `/audit/engagements/[engagementId]/trial-balance`

1. اذهب إلى تبويبة Trial Balance
2. ارفع ملف CSV أو XLSX
3. يتطلب الملف الأعمدة التالية: Account Code, Account Name, Debit, Credit, Currency
4. بعد الرفع، يظهر الجدول كاملاً مع:
   - إجمالي المدين والدائن
   - التباين (variance)
   - حالة الثقة (trust state): trusted / conditional / blocked
5. التباين غير الصفري يُبرز للمراجعة
6. SHA-256 hash يُخزّن مع الملف للتحقق من سلامته

**أنواع الملفات المدعومة:** CSV, XLSX  
**الحد الأقصى:** 20 MB

---

### 7.3 Mapping — تعيين الحسابات

**المسار:** `/audit/engagements/[engagementId]/mapping`

1. اذهب إلى تبويبة Mapping
2. كل حساب مصدري يحتاج تعييناً إلى حساب معياري (canonical account)
3. حالات التعيين:
   - **pending** — لم يتم التعيين بعد
   - **confirmed** — تم تأكيد التعيين
4. أنواع التعيين:
   - **AI suggested** — اقتراح من الذكاء الاصطناعي
   - **Manual** — تأكيد يدوي
5. التعيينات المعلقة تمنع جاهزية الاعتماد

---

### 7.4 Statements — القوائم المالية

**المسار:** `/audit/engagements/[engagementId]/statements`

1. اذهب إلى تبويبة Statements
2. تعرض ثلاث قوائم:
   - قائمة المركز المالي (Balance Sheet)
   - قائمة الدخل (Income Statement)
   - قائمة التغير في حقوق الملكية (Statement of Changes in Equity)
3. كل سطر مرتبط بتعيين حساب
4. حالات القائمة: draft → reviewed → approved
5. القوائم تُبنى تلقائياً من الحسابات المعيّنة

---

### 7.5 Notes — إيضاحات القوائم المالية

**المسار:** `/audit/engagements/[engagementId]/notes`

1. اذهب إلى تبويبة Notes
2. إيضاحات الإفصاح مرتبة بأرقام وعناوين
3. زر **"Generate Draft Notes"** يولّد إيضاحات مقترحة بالذكاء الاصطناعي
4. كل إيضاح مُنتَج بالذكاء الاصطناعي يحمل وسماً: **"AI Drafted — Not final"**
5. الإجراءات:
   - **Accept** — يحوّل المسودة إلى إيضاح حقيقي
   - **Reject** — يحذف المسودة
6. متطلبات المعلومات الناقصة تُعرض بشكل واضح

---

### 7.6 Evidence — إدارة الأدلة

**المسار:** `/audit/engagements/[engagementId]/evidence`

1. اذهب إلى تبويبة Evidence
2. ارفع ملفات الأدلة (PDF, XLSX, DOCX, JPG, PNG, CSV)
3. دورة حياة الدليل:
   - **missing** — مطلوب لكن غير موجود
   - **uploaded** — تم رفعه
   - **linked** — مرتبط بملاحظة أو كيان آخر
   - **reviewed** — تمت مراجعته
   - **accepted** — مقبول
   - **rejected** — مرفوض مع سبب
4. زر **"Suggest Evidence"** يقترح أدلة بالذكاء الاصطناعي بناءً على الأهمية النسبية والملاحظات
5. كل دليل يمكن ربطه بالملاحظات (Findings)

**الحد الأقصى:** 20 MB للملف الواحد  
**أنواع الملفات:** PDF, XLSX, XLS, DOCX, JPG, JPEG, PNG, CSV  
**يُخزّن SHA-256 لكل ملف**

---

### 7.7 Findings — الملاحظات

**المسار:** `/audit/engagements/[engagementId]/findings`

1. اذهب إلى تبويبة Findings
2. أنشئ ملاحظات جديدة أو عدّل الموجود منها
3. كل ملاحظة تحتوي:
   - العنوان (Title)
   - مستوى الخطورة (Severity): low / medium / high / critical
   - النوع (Type)
   - الحالة (Status): open / in_review / resolved
   - الوصف، السبب الجذري، الأثر
4. الملاحظات غير المحلولة ذات الخطورة العالية أو الحرجة تمنع الاعتماد
5. زر **"Suggest Findings"** ينتج ملاحظات مقترحة بالذكاء الاصطناعي

---

### 7.8 Recommendations — التوصيات

**المسار:** `/audit/engagements/[engagementId]/recommendations`

1. اذهب إلى تبويبة Recommendations
2. كل توصية مرتبطة بملاحظة محددة
3. تحتوي التوصية على:
   - مستوى المخاطرة (Risk Level)
   - الحالة (Status)
   - علامة مساهمة الذكاء الاصطناعي (AI Contribution flag)
4. التوصيات يمكن أن تكون:
   - مقترحة من الذكاء الاصطناعي (AI Drafted)
   - منشأة يدوياً
5. التوصيات المرتبطة بملاحظات مفتوحة تبقى نشطة

---

### 7.9 Review — تعليقات المراجعة

**المسار:** `/audit/engagements/[engagementId]/review`

1. اذهب إلى تبويبة Review
2. أنشئ تعليقات مراجعة تستهدف كيانات محددة:
   - **Target Entity** — اختَر من: Finding / Statement / Note
   - **Comment** — نص التعليق
   - **Required Action** — الإجراء المطلوب (revise / provide evidence / confirm)
   - **Status** — open / resolved
3. التعليقات المفتوحة تمنع الاعتماد
4. عند حل التعليق، أضف نص الحل (Resolution text)

---

### 7.10 Approval — الاعتماد

**المسار:** `/audit/engagements/[engagementId]/approval`

#### بوابة الجاهزية (Readiness Gate):

تفحص ٥ شروط قبل السماح بالاعتماد:

| # | الشرط | المعنى |
|---|--------|--------|
| 1 | جميع الحسابات معيّنة | لا تعيينات معلقة |
| 2 | الأدلة مكتملة | لا أدلة بحالة missing |
| 3 | المراجعة مكتملة | لا تعليقات مراجعة مفتوحة |
| 4 | لا ملاحظات حرجة | لا ملاحظات high/critical غير محلولة |
| 5 | الحالة مناسبة | الحالة مؤهلة للاعتماد |

- كل شرط يظهر باللون الأخضر (محقق) أو الأحمر (معطل)
- زر **Approve** معطل حتى تتحقق جميع الشروط
- زر **Reject** متاح دائماً — يمكنك رفض الارتباط مع إبداء الأسباب في أي وقت

#### عملية الاعتماد:

1. حل جميع المعطلات
2. تأكد من أن جميع الشروط خضراء
3. اضغط **Approve**
4. أدخل المبرر (Rationale)
5. يُسجّل الاعتماد مع:
   - هوية المعتمِد
   - دوره
   - الطابع الزمني
   - المبرر

---

### 7.11 Export — التصدير

**المسار:** `/audit/engagements/[engagementId]/publication`

#### أنواع التصدير:

1. **Export Financial Statements** — تصدير القوائم المالية بصيغة JSON
   - إذا كان الارتباط مسودة: يحتوي على تحذير `"DRAFT — Not final"`
   - إذا كان معتمداً: يحتوي على `approvalInfo` (المعتمِد، التوقيت، المبرر)

2. **Export Audit File** — تصدير ملف التدقيق الشامل بصيغة JSON
   - يحتوي: الأدلة، الملاحظات، التوصيات، تعليقات المراجعة، سجلات الاعتماد، مسار التدقيق

3. **Bilingual Export (AR/EN)** — تصدير ثنائي اللغة
   - عناوين القوائم بالعربية والإنجليزية
   - المحتوى المحاسبي بدون تغيير

---

### 7.12 Publication — النشر

**المسار:** `/audit/engagements/[engagementId]/publication`

1. لا يمكن النشر إلا بعد الاعتماد
2. النشر يُقفل حزمة التدقيق (lock)
3. يُسجّل الناشر والطابع الزمني
4. بعد النشر، لا يمكن تعديل محتوى الارتباط

## 8. Audit Trail & Traceability — مسار التدقيق والتتبّع

### Audit Trail
**المسار:** `/audit/engagements/[engagementId]/audit-trail`

- سجل زمني كامل لكل حدث في الارتباط
- قابل للتصفية حسب نوع الحدث
- ١٨+ نوع حدث: إنشاء، تحديث، اعتماد، رفض، ربط، تصدير، نشر...
- جميع الأحداث غير قابلة للتعديل أو الحذف (append-only)

### TraceabilityDrawer
- من أي كيان (دليل، ملاحظة، إيضاح)، اضغط **"Traceability"**
- يظهر درج منزلق يوضح:
  - **المسار الأمامي:** بيانات مصدرية → تعيين → أدلة → ملاحظات → توصيات
  - **المسار الخلفي:** نشر ← اعتماد ← مراجعة ← قوائم مالية

| افعل ✅ | لا تفعل ❌ |
|--------|------------|
| ابدأ بلوحة التحكم لإظهار الإحصائيات | لا تبدأ بالتفاصيل التقنية |
| أبرز التباين في TB | لا تتجاهل المعطلات — اشرحها |
| أظهر تعليقاً مفتوحاً وتأثيره | لا تحاول إخفاء القيود |
| نفّذ اعتماداً بعد حل المعطلات | لا تدّعِ جاهزية الإنتاج |
| اختم بمسار التدقيق وTraceability | لا تستخدم مصطلحات تقنية عميقة |

## 10. كيفية إدارة البايلوت (How to Run a Pilot)

### دورة حياة البايلوت:

| المرحلة | المدة | الأنشطة |
|---------|--------|----------|
| 1. تحضير البيئة | يوم واحد | تشغيل محلي + seed + فحص صحة |
| 2. العرض التوضيحي | ٦٠ دقيقة | جولة مباشرة على المنتج |
| 3. أسئلة ونقاش | ٣٠ دقيقة | جمع الأسئلة والملاحظات |
| 4. وصول تجريبي | ١–٢ أسبوع | استكشاف المستخدم بنفسه |
| 5. مراجعة الملاحظات | أسبوع واحد | توثيق وترتيب الأولويات |
| 6. خارطة الطريق | أسبوع واحد | جدولة بنود الإنتاج |

### خطوات جلسة البايلوت:

1. **قبل الجلسة:**
   - خذ نسخة احتياطية: `npm run db:backup`
   - شغّل فحص الصحة: `npm run audit:health`
   - تأكد من أن جميع التبويبات تستجيب
   - حضّر حساب المستخدم المناسب

2. **أثناء الجلسة:**
   - ابدأ بشرح المنتج (٣ دقائق)
   - اعرض السيناريو (٢ دقيقة)
   - نفّذ الجولة حسب Demo Script
   - سجّل أي ملاحظات مباشرة في سجل الملاحظات
   - لا تتجاهل الأخطاء — وثّقها

3. **بعد الجلسة:**
   - خذ نسخة احتياطية جديدة
   - سجّل الملاحظات في Pilot Feedback Log
   - صنّف أي مشاكل تم رصدها
   - حدّث سجل حالة البايلوت

## 12. تصنيف المشاكل (How to Classify Issues)

### Blocker — معطّل

**تعريف:** مشكلة تمنع إكمال workflow أساسي، يفشل معها البايلوت.
- التطبيق لا يعمل (خطأ 500)
- قاعدة البيانات غير متصلة
- لا يمكن حفظ البيانات
- تعطّل في ٣ workflows أو أكثر في جلسة واحدة
- **الإجراء:** أوقف الجلسة فوراً، وثّق، صعّد للمطورين. لا تكمل البايلوت حتى يُحل.

### Major Friction — احتكاك كبير

**تعريف:** workflow يعمل لكن بتجربة مستخدم سيئة جداً تجعل إكمال المهمة صعباً أو محبطاً.
- تأخير أكثر من ١٠ ثوانٍ بدون مؤشر تحميل
- رسائل خطأ تقنية غير مفهومة للمستخدم
- زر لا يعمل لكن يوجد بديل طويل
- تعطّل workflow واحد بالكامل لكن باقي الـ workflows تعمل
- **الإجراء:** وثّق، تجاوز إن أمكن، صعّد في أقرب وقت.

### Minor Polish — تحسين بسيط

**تعريف:** النظام يعمل لكن يوجد خلل في المظهر أو تجربة المستخدم.
- نص غير مترجم
- تنسيق غير متناسق
- تبويب لا يستجيب من أول نقرة
- رسالة خطأ غير مفيدة لكن مقروءة
- تأخير ١–٣ ثوانٍ
- **الإجراء:** سجّل. لا يؤثر على استمرار البايلوت.

### Product Signal — إشارة منتج

**تعريف:** ليست مشكلة تقنية بل ملاحظة من المستخدم عن ميزة مفقودة أو تحسين مطلوب.
- "أتمنى لو يوجد تصدير PDF"
- "هل يمكن أن أرسل رابطاً للعميل لرفع الأدلة؟"
- "نحتاج مقارنة بين فترتين"
- **الإجراء:** وثّق كـ product signal في ملف منفصل. لا تخلطها مع المشاكل التقنية.

## 13. ما لا يجوز تغييره أثناء البايلوت (What Not to Change During Pilot)

| ممنوع | السبب |
|--------|--------|
| ❌ Schema قاعدة البيانات | يكسر الترحيلات والـ seed |
| ❌ Routes أو مسارات جديدة | يغير سلوك التطبيق |
| ❌ Server Actions | يؤثر على جميع الـ workflows |
| ❌ آلية المصادقة | قد تمنع الدخول أثناء الجلسة |
| ❌ Seed data الحالي | يغيّر بيانات العرض |
| ❌ أي workflow أساسي | يبطل نتائج البايلوت السابقة |
| ❌ ادعاء جاهزية الإنتاج | غير دقيق — المنتج في بايلوت |

| مسموح | بشرط |
|--------|--------|
| ✅ إصلاح ترجمة | لا يؤثر على سلوك |
| ✅ إصلاح خطأ واضح في عرض البيانات | لا يمس منطق الـ workflow |
| ✅ إخفاء ميزة معطلة | عبر feature flag أو تعليق |
| ✅ توثيق | دائماً مسموح |
| ✅ تشغيل أوامر المراقبة | `audit:health`, `db:backup`, إلخ |

## 14. دليل استكشاف المشاكل (Troubleshooting Guide)

### المشكلة: التطبيق لا يعمل (`next dev` فشل)

```
الحل:
  ١. تأكد من تشغيل PostgreSQL:  تحقق من services/daemon
  ٢. تأكد من صحة DATABASE_URL في .env
  ٣. npm install && npx prisma generate && npx prisma migrate dev
  ٤. npm run dev
```

### المشكلة: خطأ "Authentication required" في الإنتاج

```
السبب: demo fallback لا يعمل في NODE_ENV=production
الحل: تأكد من تجهيز المصادقة قبل التبديل إلى وضع الإنتاج
```

### المشكلة: قاعدة البيانات لا تستجيب

```bash
npm run audit:health
# إذا فشل:
# ١. تأكد من تشغيل PostgreSQL
# ٢. تحقق من أن قاعدة البيانات aqliya موجودة
# ٣. شغّل npx prisma migrate dev
```

### المشكلة: بيانات seed غير موجودة

```bash
npm run seed:audit
npm run audit:health  # تأكد: يجب أن يظهر Engagements > 0
```

### المشكلة: صفحة بيضاء بعد الدخول

```
الحل:
  ١. افتح أدوات المطور (F12) → Console
  ٢. تحقق من عدم وجود أخطاء JavaScript
  ٣. امسح localStorage و sessionStorage
  ٤. سجّل الخروج وأعد الدخول
```

### المشكلة: زر Approve معطل رغم أن كل شيء يبدو مكتملاً

```
تحقق من بوابة الجاهزية:
  ١. هل جميع الحسابات confirmed؟
  ٢. هل توجد أدلة بحالة missing؟
  ٣. هل توجد تعليقات مراجعة مفتوحة؟
  ٤. هل توجد ملاحظات high/critical غير محلولة؟
  ٥. تأكد من أن الحالة eligible للاعتماد
```

### المشكلة: رفع ملف TB فشل

```
تحقق من:
  ١. نوع الملف: CSV أو XLSX فقط
  ٢. حجم الملف: أقل من 20 MB
  ٣. الأعمدة المطلوبة: Account Code, Account Name, Debit, Credit, Currency
  ٤. القيم الرقمية صالحة
```

### المشكلة: أداة التتبّع (Traceability) تعرض مساراً فارغاً

```
الحل:
  ١. تأكد من أن الكيان لديه روابط فعلية (مثلاً: أدلة مرتبطة بملاحظات)
  ٢. قد يكون الكيان جديداً ولم يُربط بعد
  ٣. أنشئ رابطاً يدوياً من Evidence إلى Finding ثم حاول مرة أخرى
```

## 15. عملية القرار بعد البايلوت (Post-Pilot Decision Process)

### معايير التقييم (٣٢ معياراً عبر ٦ فئات):

| الفئة | عدد المعايير | أمثلة |
|--------|---------------|--------|
| **وظيفي** | ٧ | إكمال workflow كامل < ٣٠ دقيقة، تحميل TB صحيح، بوابة اعتماد دقيقة |
| **منهجية** | ٧ | كل حدث مسجل، مسار تتبّع متكامل، AI مسودة بشرية القبول |
| **قابلية الاستخدام** | ٥ | تحميل < ٣ ثوانٍ، تنقل سلس، رسائل خطأ مفهومة |
| **أمني** | ٤ | رفض الأدوار غير المخوّلة، رفض أنواع الملفات الممنوعة |
| **تقارير** | ٤ | JSON مكتمل، علامة Draft واضحة، بيانات الاعتماد موجودة |
| **أصحاب المصلحة** | ٥ | إقرار بالقيود، قبول النطاق، ملاحظات قابلة للتنفيذ |

### آلية القرار:

| التصنيف | الحد الأدنى | المعنى |
|---------|-------------|--------|
| **GO** | ≥ ٣٠ / ٣٢ | البايلوت ناجح — الانتقال إلى تحضير الإنتاج |
| **CONDITIONAL GO** | ≥ ٢٤ / ٣٢ | نجاح مع تحفظات — معالجة الفجوات ثم GO |
| **BELOW** | ≥ ١٦ / ٣٢ | فجوات كبيرة — يحتاج جولة بايلوت أخرى |
| **NO-GO** | < ١٦ / ٣٢ | فشل — مراجعة المنتج قبل إعادة المحاولة |

### قائمة القرار النهائي:

- [ ] جميع معايير البيئة تحققت
- [ ] جميع معايير البيانات تحققت
- [ ] جميع معايير الأمان تحققت
- [ ] جميع معايير الـ workflows تحققت
- [ ] أصحاب المصلحة وافقوا على القيود
- [ ] جميع أوراق الاعتماد موقّعة
