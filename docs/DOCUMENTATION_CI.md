# AQLIYA Documentation CI Pipeline

> **Status:** Active | **Version:** 1.0 | **Date:** 2026-06-26 | **Owner:** Documentation Team

This document defines the CI pipeline for documentation validation in the AQLIYA repository. It runs on every push and pull request as part of the GitHub Actions workflow (`.github/workflows/ci.yml`).

---

## Pipeline Steps

The documentation validation runs **after `npm ci`** and **before `npm run build`** in the CI pipeline, in the following order:

| Step | Script | Purpose | Failure | 
|------|--------|---------|---------|
| 1 | `scripts/validate-documentation.mjs` | Validates knowledge-map.json entries against filesystem | Blocks |
| 2 | `scripts/validate-metadata.mjs` | Validates required metadata fields on Source of Truth documents | Blocks |
| 3 | `scripts/check-document-links.mjs` | Checks internal markdown links for broken references | Warning only |
| 4 | `scripts/find-orphan-documents.mjs` | Detects documents not referenced from any index | Blocks on unexpected orphans; warnings for known |
| 5 | `scripts/lint-documentation.mjs` | Lints for duplicate headings, empty sections, broken tables | Warning only |
| 6 | `scripts/validate-ai-consistency.mjs` | Validates cross-references between 7 AI guidance docs | Warning only |
| 7 | `scripts/validate-reading-orders.mjs` | Validates AI_ENTRYPOINT.md reading order table | Blocks |

## Local Development

Run all documentation checks locally:

```bash
npm run docs:validate       # Run all validators
npm run docs:generate-all   # Regenerate all generated docs
npm run docs:check          # Validate + health dashboard
npm run docs:health         # Regenerate health dashboard only
```

Individual validators:

```bash
node scripts/validate-documentation.mjs       # Knowledge-map validation
node scripts/check-document-links.mjs          # Link checker
node scripts/find-orphan-documents.mjs         # Orphan detector
node scripts/lint-documentation.mjs            # Documentation linter
node scripts/validate-ai-consistency.mjs       # AI consistency validator
node scripts/validate-metadata.mjs             # Metadata validator
```

Individual generators:

```bash
node scripts/generate-knowledge-map.mjs        # Regenerate knowledge-map.json
node scripts/generate-authority-matrix.mjs     # Regenerate authority matrix
node scripts/generate-health-dashboard.mjs     # Regenerate health dashboard
```

## Failure Categories

### Blocking Failures (exit code 1)

- **Knowledge-map validation** (`validate-documentation.mjs`): If any knowledge-map entry references a non-existent file, or the knowledge-map.json structure is malformed.
- **Metadata validation** (`validate-metadata.mjs`): If required metadata fields are missing from Source of Truth documents.

### Mixed (blocking or warning depending on result)

- **Orphan documents** (`find-orphan-documents.mjs`): **Blocks** if unexpected orphans are found (not in known-orphans allowlist). Non-blocking warning for known orphans listed in `docs/validation/known-orphans.json`. Review unexpected orphans and either add references, archive, or add to allowlist.

### Non-Blocking Warnings (exit code 0 with warnings)

- **Broken links** (`check-document-links.mjs`): Broken internal links are reported but do not block the build. Fix them in a follow-up.
- **Documentation lint** (`lint-documentation.mjs`): Duplicate headings, empty sections, broken tables, missing H1 on non-archive docs. Fix during documentation cleanup.
- **AI consistency** (`validate-ai-consistency.mjs`): Cross-reference gaps between AI guidance documents (exit 0 with warnings). Review and align.

## Error Output Format

All validation scripts use consistent error output:

```
  ✅ OK       — passed check
  ⚠️  Warning — non-blocking issue
  ❌ Error    — blocking failure
```

Errors and warnings include the file path and the line number (where applicable):

```
  ⚠️  docs/AGENTS.md: anchor "#section-11" not found (link: "see §11")
  ❌ docs/ai/knowledge-map.json: path "docs/nonexistent.md" not found
```

## Enforcement

The CI workflow runs documentation checks on every push and pull request. All steps must pass (or be explicitly acknowledged as warnings) before merge.

For pre-existing warnings (legacy docs), the pipeline runs without blocking the build, but warnings must not accumulate.
