## Purpose

هذا المستند يحدد البرنامج المنظم لتحويل عملاء AuditOS الأوائل (الذين أثبتوا value واضحة) إلى حسابات مرجعية (reference accounts) يمكن استخدامها في المبيعات والعروض والمحتوى المستقبلي.

## Program Design Philosophy

لا نطلب reference من كل عميل. نبنيه فقط مع العملاء الذين:

1. ظهرت لهم value حقيقية وموثقة
2. لديهم stakeholder راضٍ ومستعد للمشاركة
3. العلاقة التجارية مستقرة
4. الـ use case قابلة للعرض أو النقاش

## Reference Tiers

### Tier 1: Public Named Reference

**ما هو:** يمكن استخدام اسم العميل وشهادته علنًا.

**الشروط:**
- موافقة كتابية صريحة
- quote معتمدة من stakeholder مخولة
- العميل مقتنع بأن المشاركة لا تضره تجاريًا أو تنظيميًا

**الاستخدامات:**
- موقع AQLIYA الإلكتروني
- sales decks
- outbound materials
- social content
- investment conversations

---

### Tier 2: Anonymized Case Study

**ما هو:** تُروى الحالة بدون ذكر اسم العميل، مع وصف القطاع وحجم الفريق ونوع use case.

**الشروط:**
- موافقة كتابية على مشاركة التفاصيل بدون الاسم
- التأكد من أن التفاصيل لا تكشف هوية العميل
- مراجعة العميل للمسودة قبل النشر

**الاستخدامات:**
- sales decks
- proposal attachments
- founder updates
- website case studies section
- outbound proof points

---

### Tier 3: Private Reference Call

**ما هو:** العميل يوافق على التحدث مع prospects محددين عبر call خاص.

**الشروط:**
- موافقة العميل على هذا النوع من المشاركة
- تحديد عدد المرات أو الفترة الزمنية (لتجنب إرهاق العميل)
- يتم تنسيق كل call بموافقة مسبقة

**الاستخدامات:**
- late-stage pipeline
- enterprise prospects
- شركات مشابهة في القطاع أو الحجم

---

### Tier 4: Metric-Only Proof

**ما هو:** استخدام metrics أو signals كمية بدون أي إشارة للعميل.

**الشروط:**
- metrics موثقة من pilot/paid data
- لا يمكن تتبعها إلى عميل محدد
- لا تحتوي على أي سياق يكشف الهوية

**الاستخدامات:**
- sales decks
- one-pagers
- outbound emails
- public content

---

### Tier 5: No Reference

**ما هو:** العميل لا يشارك بأي شكل.

**القاعدة:** يُحترم هذا القرار تمامًا. لا متابعة. لا ضغط.

## Reference Request Workflow

```
Pilot closeout / paid confirmation
    → referenceability decision
    → إذا كان العميل مؤهلًا
        → اختيار tier المناسب
        → طلب الموافقة حسب tier
        → توثيق الموافقة
        → إضافة إلى proof asset index
```

## When To Ask

| التوقيت | الإجراء |
|---------|---------|
| بعد pilot closeout مباشرة | تقييم referenceability الأولي |
| بعد أول paid period ناجحة | أفضل توقيت لطلب reference أعلى |
| بعد تجديد أو توسع | فرصة لتحديث الـ reference |
| أبدًا أثناء pilot | لا تطلب reference قبل value مثبتة |

## What To Offer In Return

لا تشتري references. لكن يمكن تقديم:

1. أولوية في roadmap requests (ضمن حدود معقولة)
2. ذكر اسم الشركة في case study (إذا كان هذا ذا قيمة للعميل)
3. علاقة أوثق مع فريق AQLIYA
4. الوصول المبكر لميزات جديدة

## Reference Account Health

لكل reference account تابع:

1. آخر مرة استُخدم فيها كمرجع
2. عدد الـ reference calls المنجزة
3. حالة رضا العميل الحالية
4. هل يحتاج التقييم إلى تحديث؟

## Program Limits (First 3–5 Customers)

مع أول 3-5 عملاء:

1. لا نضغط على أي عميل ليكون reference
2. نكتفي بـ 1-2 references موثوقين
3. نفضّل quality على quantity
4. كل reference يجب أن يعكس value حقيقية وليس hype

## What Not To Do

1. لا تطلب reference قبل أن يقتنع العميل داخليًا بالقيمة
2. لا تضغط على stakeholder متحفظة
3. لا تستخدم اسم العميل بأي شكل بدون موافقة صريحة
4. لا تفرط في استخدام نفس العميل المرجعي
5. لا تبنِ case study على claims غير مثبتة
