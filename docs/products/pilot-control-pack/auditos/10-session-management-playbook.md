# AuditOS Pilot Session Management Playbook

## Session Lifecycle

```
1. Qualify → 2. Schedule → 3. Prepare → 4. Execute → 5. Capture → 6. Review → 7. Follow-up
```

---

## Phase 1: Qualify

**Input:** Candidate list (09)

**Actions:**

1. Confirm candidate meets must-have criteria
2. Send one-paragraph value proposition (not a spec sheet)
3. If interested, schedule a 15-min intro call
4. On the intro call:
   - Confirm they understand "controlled pilot" (not production)
   - Confirm they have or accept demo data
   - If they ask for production: politely decline and explain the pilot scope
5. If qualified, proceed to scheduling

**Gate:** If candidate does not understand "controlled pilot" after explanation, do not proceed.

## Phase 2: Schedule

**Rules:**

- Sessions are 60 minutes max (45 min workflow + 15 min buffer)
- Sessions are founder-led only
- No more than 2 sessions per week (founder bandwidth)
- At least 48h between sessions (for preparation and recovery)
- Sessions are recorded (with participant consent) or documented in real-time

**Calendar invite includes:**

- Link to constraints doc (08) — must be read before session
- Duration: 60 min
- Format: Video call + shared screen
- What to prepare: None (demo data provided)

## Phase 3: Prepare

**Before each session (60 min):**

1. Review candidate profile and notes
2. Complete environment checklist (02)
3. Verify seed data is fresh
4. Rehearse the runbook (06) if needed
5. Open evidence capture template (05)
6. Prepare screenshots of key screens (in case of connectivity issues)
7. Review any feedback from previous sessions that might apply

**Emergency prep:**

- Have rollback procedure (03) accessible
- Have escalation template (04) ready
- Know how to reset environment quickly

## Phase 4: Execute

**Follow the runbook (06). Do not skip phases.**

Key disciplines:

- Start with constraints briefing
- Let the participant drive where possible (screen control)
- Narrate what you're doing and why
- Pause when participant looks confused
- Capture observations in real-time (not from memory after)
- If something breaks, name it, log it, and decide: continue or abort?
- Do NOT promise fixes during session. Say: "I've noted that. We'll review it."

## Phase 5: Capture

**Immediately after session (within 30 min):**

1. Complete evidence capture template (05)
2. Save screenshots to session folder
3. Note any incidents in escalation template (04)
4. Save audit logs relevant to the session

**Do NOT:**

- Rely on memory — write it down immediately
- Edit participant's words — capture verbatim
- Delete any session data without explicit approval

## Phase 6: Review

**Within 24 hours of session:**

1. Complete post-session review (07)
2. Extract action items
3. Classify feedback:
   - **Blocker:** Must fix before next session
   - **Friction:** Should fix before broader rollout
   - **Nice-to-have:** Log for later
   - **Misunderstanding:** Update runbook/constraints
4. Update candidate tracking table (09)

**Review meeting (weekly, with all sessions):**

- What did we learn?
- What broke?
- What did participants trust/distrust?
- Should we adjust the pilot scope?

## Phase 7: Follow-up

**Within 48 hours of session:**

1. Send thank-you email to participant
2. Include summary of what was demonstrated
3. Do NOT include commitments or timelines
4. Ask for any additional thoughts they've had since
5. If they want a second session, schedule after review is complete

**After second session (if applicable):**

- Consider moving to "proof" phase (11)
- Ask about referral / case study interest
- Do NOT ask for purchase intent before 3 sessions

## Session Cadence

| Week | Sessions | Focus                                                      |
| ---- | -------- | ---------------------------------------------------------- |
| 1    | 1–2      | Validate runbook, fix blockers                             |
| 2    | 2–3      | Refine messaging, collect evidence                         |
| 3    | 2–3      | Begin proof conversion                                     |
| 4    | Review   | Assess, decide: continue pilot or begin controlled rollout |

## Founder Discipline

| Do                            | Don't                                        |
| ----------------------------- | -------------------------------------------- |
| Lead every session personally | Delegate to non-technical team member        |
| Listen more than you talk     | Pitch features instead of watching reactions |
| Capture verbatim feedback     | Interpret or soften feedback                 |
| Follow the runbook            | Improvise new workflows during session       |
| Log every issue               | Fix issues during session                    |
| End on time                   | Go overtime (participant loses focus)        |
| Thank the participant         | Pressure for commitment                      |
